# Runner Core v1

Status: v1.1.0 implementation guide  
Applies to: `bin/basics.js`

This document explains exactly how the BASICS runner works today, what checks it performs, how it makes decisions, and where the current boundaries are.

## 1) What the runner is

The runner is a local CLI execution engine that:

1. parses a command
2. discovers and scans a target repository
3. loads a rule pack
4. evaluates each rule
5. computes blocking status
6. writes machine and human reports
7. exits with deterministic status codes

The implementation is intentionally dependency-light (Node standard library only).

## 2) Core execution model

Entry point:

- `bin/basics.js` (`main()`)

High-level command dispatch:

- `init`
- `detect`
- `dirty-test`
- `assess`
- `report`
- `claim:validate`
- `rules:list`
- `version`

The runner uses a single executable and branches behavior by command + selected rule pack.

## 3) Command parsing and process behavior

### 3.1 Argument parsing

Function:

- `parseArgs(argv)`

Behavior:

- collects positional command in `args._[0]`
- collects flags by `--key value` or `--key` boolean
- does not currently support nested/complex flag parsing

### 3.2 Exit codes

Constants in `EXIT`:

- `0` success
- `10` configuration issue
- `20` target discovery issue
- `30` evaluation issue (reserved path)
- `40` blocking conformance findings
- `50` unexpected runtime failure

Blocking findings in assessment commands force exit `40`.

## 4) Data and report entities

The runner materializes a report object with:

- `run`: run metadata (id, mode, target, tier, profiles, timestamp, toolVersion, rulePack)
- `summary`: counts by status + blockingFailures
- `results`: one object per rule (`ruleId`, `status`, `severity`, `observed`, `evidence`, `remediation`)
- `confidence`: confidence level + rationale
- `limits`: explicit declared limitations

Run IDs are generated by:

- `toRunId(mode, target, tier)`

## 5) Repository discovery and profiling

### 5.1 File scanning

Function:

- `listFilesRecursive(dir)`

Skips:

- `.git`
- `node_modules`
- `.basics`

Purpose:

- keep scans deterministic and reasonably fast

### 5.2 Repo type detection

Function:

- `detectRepoType(files)`

Heuristics:

- CLI signals: `cli.md`, `package.json`, common entry filenames
- web/app signals: `index.html`, common web config names

Outputs:

- `cli`, `app`, `mixed`, `unknown`

### 5.3 Profile inference

Function:

- `inferProfiles(targetPath, files)`

Heuristics:

- base: `shared-core`
- add `software` if package or python project manifest found
- add `hardware`/`firmware` by filename signals
- fallback includes `software` if no clear signal

## 6) Rule pack system

Current packs:

- `rules/dirty-core-v1.json`
- `rules/assess-core-v1.1.json`

Pack loading functions:

- `loadDirtyPack()`
- `loadAssessPack()`
- `loadPackByMode(mode)`
- `loadPackById(packId)`

Mode mapping:

- `dirty-test` -> dirty pack
- `assess` -> assess pack

## 7) Rule evaluation engine

Single evaluator dispatcher:

- `evaluateRule(ruleId, ctx)`

This function contains explicit evaluators per supported rule ID.
If a rule has no evaluator, result status is `error`.

### 7.1 Evaluator strategy types used

1. file existence checks
2. content keyword checks
3. code-pattern signal scans
4. partial scoring logic for incomplete evidence

### 7.2 Rule output helpers

- `pass(...)`
- `fail(...)`
- `partial(...)`

All evaluators return normalized result objects so summarization and report rendering stay generic.

## 8) Implemented rule behavior (v1.1.0)

### Evidence rules

- `BASICS-EVID-001`: checks for command surface artifact (`cli.md`, `protocol/cli.md`, `COMMANDS.md`)
- `BASICS-EVID-002`: checks for event schema artifact (`event-schema.md`/schema paths)
- `BASICS-EVID-003`: checks for ADR index artifact (returns partial if missing)
- `BASICS-EVID-004`: checks both compatibility policy and deviation registry artifacts
- `BASICS-EVID-005`: checks degraded-mode matrix and evidence location

### Shared core behavioral rules

- `BASICS-SC-001`: verifies command doc exists and appears semantically complete
- `BASICS-SC-012`: checks canonical command-class keyword coverage in command docs
- `BASICS-SC-041`: heuristic code scan for create/edit + local persistence signals
- `BASICS-SC-050`: requires both command semantics artifact and event schema artifact
- `BASICS-SC-051`: checks compatibility policy for additive/breaking/migration language

### Tier rule

- `BASICS-TIER-010`: aggregate gate requiring command docs, event schema, compatibility policy, degraded matrix, and local create/edit signal

## 9) Blocking and strictness logic

Function:

- `summarize(results, strict, rulesMeta)`

Blocking by default:

- mandatory rule with `fail`
- mandatory rule with `error`

Additional blocking in strict mode:

- mandatory rule with `partial`

This gives a useful gradient:

- non-strict = practical early adoption
- strict = near-gate review pressure

## 10) Report rendering and output

### 10.1 Renderers

- JSON report is direct structured output
- Markdown report produced by `renderMarkdown(report, rulesMeta)`

### 10.2 Output writing

Function:

- `writeRunOutputs(report, markdown, outDir)`

Writes:

- `basics-report-<runId>.json`
- `basics-report-<runId>.md`
- `basics-summary-<runId>.txt`

### 10.3 Direct report write

Flag:

- `--write-report <path>`

Behavior:

- writes markdown report directly to requested file path
- still writes standard artifacts under output directory

## 11) Claim validation path

Command:

- `claim:validate --file <claim.json|claim.yaml|claim.yml>`

Parsing:

- JSON path uses native JSON parse
- YAML path uses built-in simple YAML parser (`parseSimpleYaml`)

Required fields checked:

- `product`
- `basicsVersion`
- `tier`
- `profiles`
- `evidenceUri`

Tier check:

- must be one of `core|field|industrial`

## 12) Rules listing path

Command:

- `rules:list [--mode dirty-test|assess]`

Output includes:

- `toolVersion`
- pack id/version
- rule list with severity, mandatory flag, and profiles

## 13) Current limitations (important)

1. Rule evaluation is rule-id hardcoded; no plugin loader yet.
2. Code checks are heuristic pattern scans, not AST-level semantic analysis.
3. `assess` mode uses expanded pack but still same evaluator architecture as dirty mode.
4. YAML parser is intentionally simple and supports only basic key/list patterns.
5. No field/industrial tier-specific evaluator set yet.
6. No remote evidence URI verification in claim validation.
7. No built-in trend/diff mode between runs.

## 14) Why this core design is useful

- It is deterministic enough to pressure-test repos now.
- It is strict enough to produce uncomfortable, actionable findings.
- It produces both machine and human outputs for tooling and governance.
- It stays simple enough to evolve quickly without framework lock-in.

## 15) Immediate extension points

1. Add evaluator registry by rule metadata type.
2. Add field/industrial packs with distinct gates.
3. Add rule-to-source citation embedding in report output.
4. Add report diff command (`basics diff-report`).
5. Add CI mode (`--ci`) with concise summary output.
